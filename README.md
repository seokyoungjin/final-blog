## 🤔 1. 왜 이 프로젝트를 만들었는가?

단순히 기능을 구현하는 연습이 아니라,  
**실제 서비스와 비슷한 형태의 블로그를 처음부터 끝까지 직접 만들어보고 싶었습니다.**

React.js를 많이 사용한다고는 알고 있었지만,

-   왜 많이 쓰이는지
-   어떤 상황에서 쓰는 것이 좋은지
-   장점과 단점은 무엇인지

**직접 하면서 경험하고 알아갔습니다.**

많은 시행착오가 있었지만 하나의 서비스 형태로 직접 만들어보며 이해해보고자 했습니다.

또한 백엔드에서 데이터를 fetch로 가져와 화면에 보여주는 과정을 경험하면서,  
**프론트엔드가 단순히 화면만 그리는 것이 아니라  
데이터와 함께 동작하는 역할이라는 것을 직접 느껴보고 싶었습니다.**

## ⚛️ 2. 왜 React를 사용했는가?

이 프로젝트에서 가장 중요하게 생각한 점은 UI 상태 관리와 사용자 경험(UX)이었습니다.

### React를 선택한 이유

#### 1) 프론트엔드 개발자의 필수 기술이기 때문이다

현업에서 가장 많이 사용되는 라이브러리이며, 생태계가 풍부하고 레퍼런스가 많습니다.

#### 2) 컴포넌트 단위로 UI를 분리해 유지보수성과 재사용성을 높일 수 있음

기능 단위로 컴포넌트를 나누면 코드의 역할이 명확해지고, 수정 시 영향 범위를 줄일 수 있습니다.

#### 3) 상태 기반 UI 렌더링

React에서는 상태만 관리하면 화면은 그 상태에 맞게 자동으로 업데이트되도록 구성할 수 있습니다.  
이는 복잡한 UI 로직을 단순하게 만들어 줍니다.

## 🔍 3. 문제 인식과 해결 접근

프로젝트를 진행하면서 마주한 문제들과 해결 과정을 기록했습니다.

### 1) 컴포넌트가 커질수록 구조가 복잡해지는 문제

#### 문제 상황

페이지 단위로 컴포넌트가 커지면서  
렌더링 흐름과 상태 관리가 한눈에 보이지 않는 문제가 있었습니다.

#### 해결 접근

-   화면 단위 컴포넌트를 기능 기준으로 나누기
-   공통으로 사용되는 UI는 별도의 컴포넌트로 분리

#### 결과

각 컴포넌트의 역할이 명확해지고, 수정 시 영향 범위를 줄일 수 있었습니다.

<br/>

### 2) 불필요한 리렌더링으로 인한 성능 저하 가능성

#### 문제 인식

상태 변경이 많아지면서  
필요하지 않은 컴포넌트까지 다시 렌더링될 수 있다는 점을 인식했습니다.

#### 해결 접근

-   상태를 최소한의 범위에서만 관리
-   화면에 직접 영향을 주는 데이터만 상태로 두는 방향을 고민

#### 현재 상태

아직 완벽한 최적화 단계는 아니지만,  
**렌더링이 왜 다시 발생하는지 의식하면서 코드를 작성**하는 계기가 되었습니다.

<br/>

### 3) 사용자 흐름 기준 UX 불편 문제

#### 문제 상황

글 작성 → 목록 → 상세 페이지로 이동하는 과정에서  
사용자 입장에서 **동작 흐름이 끊긴다고 느껴지는 부분**이 있었습니다.

#### 해결 접근

-   페이지 이동 시 **불필요한 화면 전환을 줄이기**
-   사용자가 **"지금 어떤 상태인지" 알 수 있도록** 화면 구성에 신경 쓰기

#### 인사이트

기능보다 **사용자 입장에서 자연스러운 흐름인지**를 먼저 생각하려고 했습니다.

<br/>

### 4) API 요청 실패 시 사용자 피드백 부족 문제

#### 문제 상황

API 요청이 실패했을 때  
**화면이 비어 있거나 아무 반응이 없는 문제**가 발생할 수 있다는 점을 인식했습니다.

#### 해결 접근

-   데이터가 없을 경우를 항상 가정
-   **로딩 상태와 오류 상태를 분리**해서 처리

#### 목표

사용자가 **이유를 알 수 없는 화면을 마주하지 않도록** 하는 것

<br/>

> **문제를 그냥 넘기지 않고,  
> 구조와 사용자 관점에서 바라보는 연습을 한 프로젝트입니다.**

## 🏗️ 4. 사용한 기술과 아키텍처

### 📁 컴포넌트 구조

```
src/
├── components/       # 공통 UI 컴포넌트
│   ├── Button.tsx
│   ├── Layout.tsx
│   └── ...
├── pages/           # 페이지 단위 컴포넌트
│   ├── Home.tsx
│   ├── PostDetail.tsx
│   └── ...
├── hooks/           # 커스텀 훅
├── store/           # 상태 관리 (Zustand)
├── services/        # API 통신
└── types/           # TypeScript 타입 정의
```

#### 설계 원칙

-   **UI 컴포넌트와 비즈니스 로직을 분리**
-   공통 컴포넌트(Button, Layout 등)는 **재사용 가능하도록 구성**
-   **페이지 단위 컴포넌트**와 **기능 단위 컴포넌트**를 명확히 구분

<br/>

### 🔄 데이터 흐름

```
1. 사용자 액션 발생 (글 조회, 이동 등)
   ↓
2. API 요청
   ↓
3. 상태 업데이트 (Zustand)
   ↓
4. 상태 변화에 따른 UI 자동 반영
```

React의 **단방향 데이터 흐름**을 기반으로  
**데이터 → 상태 → UI의 흐름이 명확하게 보이도록** 설계했습니다.
